# Портал объектов — техническая справка для разработчика

Внутренняя система ВПК «Прогресс Энерго» для ведения проектно-изыскательских работ (ПИР):
объекты, разделы проектной/рабочей документации, сроки, исполнители, вопросы, ИРД, экспертиза.

> Документ для разработчика, который принимает систему на сопровождение. Пользовательская
> инструкция — отдельный файл `Инструкция пользователя — Портал объектов.docx`.

---

## 1. Архитектура (коротко)

```
Браузер сотрудника
      │  (GitHub Pages, HTTPS)
      ▼
index.html  ── один файл: HTML + CSS + ванильный JS, без сборки и фреймворков
      │  fetch GET/POST
      ▼
https://cg156688.tw1.ru/api.php   ── PHP-прокси на хостинге Timeweb (российский сервер)
      │  PDO
      ▼
MySQL, таблица storage(k, v)      ── хранилище «ключ → значение» (значение = JSON-строка)
```

- **Фронтенд** — статический файл на GitHub Pages. Никакого билда: открыл `index.html` — там всё.
- **Бэкенд** — крошечный PHP-файл `api.php` поверх MySQL. Реализует key/value API.
- **Почему российский хостинг:** ранее бэкенд был на Supabase, затем за Cloudflare Worker —
  у части офисов провайдеры рвали HTTP/2/HTTP/3 к зарубежным хостам
  (`ERR_HTTP2_PING_FAILED`, `ERR_QUIC_PROTOCOL_ERROR`). Перенос на российский Timeweb решил это.
  В коде остались неиспользуемые `SB_KEY/SB_H/SB_Q` — артефакты прежней схемы, можно удалить.

## 3. Бэкенд `api.php`

Таблица создаётся автоматически: `storage(k VARCHAR(191) PRIMARY KEY, v LONGTEXT)`.
В начале файла — доступы к MySQL (`$DB_HOST/$DB_NAME/$DB_USER/$DB_PASS`). Отдаёт CORS (`*`).

| Запрос | Действие |
|--------|----------|
| `GET ?key=ИМЯ` | вернуть одну запись: `[{ "key", "value" }]` |
| `GET ?keys=a,b,c` | вернуть несколько записей по списку ключей |
| `GET ?prefix=piron_obj_` | вернуть все записи, чей ключ начинается с префикса |
| `POST` тело `{ "key", "value" }` | upsert (INSERT … ON DUPLICATE KEY UPDATE) |
| `DELETE ?key=ИМЯ` | удалить запись |
| `OPTIONS` | preflight → 204 |

`value` всегда строка (JSON, сериализованный на клиенте). Сервер в содержимое не вникает.

## 4. Модель данных (ключи в `storage`)

| Ключ | Содержимое |
|------|-----------|
| `piron_users` | массив пользователей |
| `piron_obj_<id>` | один объект (основное хранение, по одному ключу на объект) |
| `piron_objects` | устаревший общий массив объектов (legacy, поддерживается для совместимости) |
| `piron_backoffice` | массив «бэкофисных» вопросов (вне объектов) |
| `piron_executors` | массив исполнителей/подрядчиков |
| `piron_sessions` | карта активных сеансов (для блокировки «один вход») |
| `piron_<fileId>` | загруженные файлы (data-URL), по требованию |

### Объект
```js
{
  id, _del,                       // _del = «архивирован» (мягкое удаление)
  passport: { name, ... },
  curated,                        // true → учитываются только разделы с on===true
  gipId, podryadId, pomGipId,     // назначенные ответственные (id пользователей)
  team: [userId, ...],            // команда объекта
  rows: {                         // разделы (ключи p1..p12 — Стадия П; марки — Стадия Р)
    p1: { st, who, start, end, on, files:{task,tpl,fin}, exonSt, history:[...] }
  },
  ird: { i_egrn: { st, on, ... }, ... },   // исходные данные
  tasks: [ { id, title, sec, from, prio, status, due, assignee, author, created, _del } ],
  extra: { P:[], R:[], I:[] }     // кастомные подразделы/ИРД (ключи pc*/rc*/ic*)
}
```

### Пользователь
```js
{ id, login, password, name, role, company, phone, email, _del }
```
`password` хранится в открытом виде (внутренний инструмент, доступ по anon-ключу). При повышении
требований — заменить на хеш и серверную проверку в `api.php`.

### Справочники-константы (в `index.html`)
- `STAGE_P` — разделы Стадии П (ПП РФ № 87), `STAGE_R` — марки Стадии Р (ГОСТ Р 21.101-2026), `IRD` — состав ИРД.
- `STATUSES`/`STAT_COLOR` — статусы разделов и цвета; `IRD_STATUSES`; `TK_STATUSES` — статусы вопросов.
- `ROLE_NAME`/`ROLE_LIST` — роли. `TASK_FROM`/`FROM_COLOR` — источники вопросов.

## 5. Фронтенд `index.html` — как устроен

Состояние в памяти: `let DB = { users, objects, backoffice, executors }` + `currentUser`, `view`, `activeTab`.
Кэш в `localStorage`: `piron_db_v3`, `piron_session`.

Перерисовка — функция **`render()`** собирает HTML строкой и кладёт в `#root`. Экраны (`view`):
`login`, `portfolio` (Объекты), `object` (карточка объекта), `lk` (Личный кабинет), `tasks`
(Все задачи), `backoffice`, `execs` (Исполнители), `users`, `archive`. Вкладки объекта (`activeTab`):
дашборд, Стадия П, Стадия Р, ИРД, Схема (`renderSchema`), График (`renderGantt`), Экспертиза, Команда.

### Синхронизация (важно)
- **`fetchRemote()`** — каждые 60 c: тянет `piron_users/objects/backoffice/executors` + все `piron_obj_*`,
  сливает с локальными через **`mergeById()`** (объединение по `id`; при конфликте берётся серверная
  запись; локальные, которых нет на сервере, сохраняются — принцип «ничего не теряем»).
- **`pushKey(key, data)`** — upsert одного ключа на сервер (debounce 900 мс) + `saveLocal()`.
  Обёртки: `persistOneObj()`, `persistUsers()`, `persistBO()`, `persistExec()`.
- **`bootSeed()`** — при старте локально создаёт `admin/admin`, чтобы система не блокировалась,
  если бэкенд недоступен; временная запись `id:'seed-admin'` удаляется, когда придут реальные данные.

### Права доступа
- `visibleObjects()` — admin/director/advisor/rp видят все; gip/pom_gip/podryad/smetchik — только «свои» (`isMember()`).
- `canEdit()` — admin и rp правят всё; director/advisor — только просмотр; остальные — свои объекты (с учётом scope).
- `canAsk()`, `canBO()` — право задавать вопросы / вести бэкофис.

### Вход и «один сеанс»
- `doLogin()` сверяет логин/пароль (логин — регистронезависимо и с trim; пароль — с trim).
- Один активный сеанс на учётку: токен в `piron_sessions`, «сердцебиение» каждые 30 c,
  `SESSION_TTL = 120000` мс.

### Демо-режим (`demo.html`)
`const DEMO=true` → `fetchRemote`/`getSessions` отключены, `pushKey` пишет только в localStorage
(ключи `pirondemo_*`), `seedDemo()` создаёт вымышленные объекты/пользователей, внизу баннер.

## 6. Развёртывание

**Фронтенд (GitHub Pages):** правки в `index.html` → commit → push в `main` репозитория
`VPK-ProgressEnergy/Portal`. Pages пересобирается за 1–2 минуты (новые файлы появляются чуть позже).
Браузерный кэш сбрасывается добавлением `?v=…` к URL или `Ctrl+F5`.

**Бэкенд (`api.php`):** через файловый менеджер Timeweb в `vpk-portal/public_html/`.
При смене базы/доступов — править переменные вверху `api.php`.

**Локальный запуск:** достаточно открыть `index.html` (или поднять любой статический сервер,
напр. `python -m http.server`). Данные всё равно идут на боевой `api.php` (см. `SB_URL`), поэтому
для экспериментов используйте `demo.html` либо поднимите свой `api.php` и поменяйте `SB_URL`.

## 7. На что обратить внимание

- ⚠️ **Хостинг Timeweb сейчас на тестовом периоде** — без оплаты бэкенд и адрес `cg156688.tw1.ru`
  отключатся. Технический поддомен можно заменить на полноценный (тогда поменять `SB_URL`).
- Единая точка конфигурации адреса бэкенда — константа **`SB_URL`** в начале `<script>`.
- Безопасность: пароли в открытом виде, API открыт по сети (как и раньше с anon-ключом). Для
  ужесточения — хеши паролей + токен/проверка в `api.php`.
- Кодовая база намеренно «в один файл» — для лёгкого хостинга и правок без сборки. Менять
  лучше точечно, проверяя в `demo.html`/локально, и не вызывая `pushKey`/`persist*` на тестовых
  данных против боевого `api.php`.
